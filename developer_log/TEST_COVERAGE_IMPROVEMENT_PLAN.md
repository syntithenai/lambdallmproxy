### Test Naming Conventions

**Unit Tests**:
- File: `tests/unit/<module-path>/<module-name>.test.js`
- Example: `tests/unit/services/user-isolation.test.js`

**Integration Tests**:
- File: `tests/integration/<feature-name>.test.js`
- Example: `tests/integration/multi-tenancy.test.js`

**Test Structure**:
```javascript
describe('ModuleName', () => {
  describe('functionName', () => {
    test('should handle happy path', () => { /* ... */ });
    test('should handle error case', () => { /* ... */ });
    test('should validate input', () => { /* ... */ });
  });
});
```

## Testing Best Practices

### 1. Test Isolation
- ‚úÖ Use `beforeEach` to reset state
- ‚úÖ Mock all external dependencies
- ‚úÖ No shared state between tests
- ‚úÖ Clean up resources in `afterEach`

### 2. Mock Strategy
- ‚úÖ Mock AWS SDK (already in `tests/mockAwsSdk.js`)
- ‚úÖ Mock Google Sheets API
- ‚úÖ Mock LLM provider APIs (OpenAI, Groq, Gemini)
- ‚úÖ Mock external services (YouTube, Tavily, PayPal)
- ‚úÖ Use fixtures for consistent test data

### 3. Coverage Targets
- üéØ **Critical Modules**: 90%+ coverage
  - Authentication, billing, multi-tenancy, core handlers
- üéØ **Important Modules**: 80%+ coverage
  - Endpoints, tools, services
- üéØ **Utilities**: 70%+ coverage
  - Helper functions, formatters

### 4. Test Types
- **Unit Tests**: Fast, isolated, mock all dependencies
- **Integration Tests**: Slower, test component interactions
- **E2E Tests**: Manual testing checklist (see below)

### 5. Continuous Integration
- ‚úÖ Run tests on every commit (via Husky pre-commit hook)
- ‚úÖ Block merges if tests fail
- ‚úÖ Generate coverage reports
- ‚úÖ Track coverage trends over time
